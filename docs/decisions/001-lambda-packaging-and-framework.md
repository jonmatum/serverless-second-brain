# ADR-001: Lambda Packaging (Zip) and No Web Framework

**Status**: Accepted
**Date**: 2026-03-20
**Context**: Issue #1 (foundation), `.kiro/steering/architecture.md`

## Decision

Use zip-packaged Node.js 22.x Lambda functions with direct API Gateway integration. No web framework (FastAPI, Express, Hono) inside Lambda handlers.

## Context

Before starting Phase 1 implementation, we evaluated two alternatives:

1. **Container images (ECR)** instead of zip packages
2. **FastAPI (Python)** as a framework inside Lambda

## Options considered

### Packaging: Zip vs Container Image

| Factor | Zip | Container Image |
|---|---|---|
| Cold start | ~200-500ms | ~500-1500ms |
| Package size limit | 250 MB | 10 GB |
| Local dev | `node index.mjs` or SAM CLI | `docker run` |
| CI/CD | Zip + upload | Build → push ECR → update Lambda |
| Idle cost | $0 | ECR ~$0.10/GB/mo per image |
| Dependency needs | AWS SDK + shared types (~5 MB) | Useful for native binaries, large deps |

Our Lambdas are small TypeScript functions with minimal dependencies (AWS SDK v3 tree-shaken, shared types). No native binaries, no large ML models. Zip packaging is sufficient and keeps cold starts at the lower end.

### Framework: None vs FastAPI vs Hono

| Factor | No framework | FastAPI (Python) | Hono (Node.js) |
|---|---|---|---|
| Cold start overhead | 0ms | +300-500ms (Pydantic, Mangum) | +50-100ms |
| Routing | API Gateway routes to individual Lambdas | Redundant — two routers (APIGW + FastAPI) | Redundant — two routers |
| Language | TypeScript (matches jonmatum.com stack) | Python (splits the codebase) | TypeScript |
| Auth/throttling | API Gateway handles it | Must configure in both layers | Must configure in both layers |
| OpenAPI generation | From steering file `api-spec.md` | Auto-generated (a FastAPI strength) | Manual |
| Adapter needed | No | Yes (Mangum for ASGI→Lambda) | Yes (adapter for Lambda events) |

FastAPI's strengths (auto OpenAPI, dependency injection, middleware) are redundant when API Gateway already provides routing, auth, throttling, and CORS. The ASGI adapter adds cold start latency for features we don't use.

A lightweight framework like Hono would add less overhead but still introduces a redundant router. With 4 Lambdas each handling a single route, there's nothing to route inside the function.

## Decision rationale

1. **Cost constraint**: $0.51/mo idle target requires minimal infrastructure. ECR storage and container cold starts work against this.
2. **Simplicity**: one handler per Lambda, one route per handler. API Gateway is the router. No framework overhead.
3. **Cold starts**: zip + Node.js 22.x gives the fastest cold starts (~200ms), critical for a low-traffic personal system where most invocations are cold.
4. **Language consistency**: TypeScript across all Lambdas and the existing jonmatum.com stack. Python agents (Strands) are a separate concern in a separate repo.
5. **SDD approach** (see ADR-009): API contracts are defined in `api-spec.md` steering file, not auto-generated by a framework. The spec drives the code, not the other way around.

## Consequences

- Lambda handlers are plain TypeScript functions that receive API Gateway events and return responses
- Shared code (DynamoDB client, error types, Bedrock client) lives in `src/shared/`
- OpenAPI spec will be generated from `api-spec.md` if needed, not from framework introspection
- If a future Lambda needs native binaries or exceeds 250 MB, we can migrate that specific function to a container image without changing the others
- If AgentCore Runtime (Phase 3, issue #8) needs a persistent server, FastAPI or Hono could be reconsidered for that specific service
